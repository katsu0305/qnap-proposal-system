## 1. エグゼクティブサマリー

### 1.1 プロジェクト概要

QNAP NAS製品の提案業務を効率化し、営業担当者が専門知識なしでも高品質な提案を行えるWebベースの支援ツールです。

```
┌─────────────────────────────────────┐
│  営業担当者の課題                    │
│  ・製品選定に専門知識が必要          │
│  ・見積もり作成に時間がかかる        │
│  ・提案品質が担当者によってばらつく  │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  QNAP提案支援システム                │
│  ・ガイド付き製品選定                │
│  ・自動見積もり計算                  │
│  ・標準化された提案資料              │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  期待される効果                      │
│  ・見積もり作成時間: 30分 → 10分    │
│  ・提案品質の均一化                  │
│  ・新人営業の立ち上がり期間短縮      │
└─────────────────────────────────────┘
```

### 1.2 現状の課題

#### 営業現場で起きている問題

| 課題 | 具体的な影響 | 頻度 |
|------|------------|------|
| **製品選定に時間がかかる** | 5製品×複数の構成パターンから選ぶ必要がある | 毎回 |
| **見積もり計算が複雑** | HDD容量・RAID構成・保守プランの組み合わせ計算 | 毎回 |
| **提案品質のばらつき** | ベテランと新人で提案内容に差が出る | 常時 |
| **専門知識の習得に時間** | RAID構成、ネットワーク知識などの理解が必要 | 新人時 |
| **資料作成の手間** | 毎回、提案書を一から作成 | 毎回 |

**実例**: 
- 中規模企業への提案で、製品選定に2時間、見積もり作成に1時間、合計3時間を要するケースが頻発
- 新人営業は製品知識不足により、提案機会を逃すケースあり

### 1.3 解決策: QNAP提案支援システム

本システムは、営業担当者が**ブラウザ上で簡単な質問に答えるだけ**で、最適な製品構成と見積もりを自動生成します。

**3つの核心価値**:


## 2. 営業現場の課題(詳細)

### 2.1 ビフォー: 現状の問題点

#### 🔴 問題1: 製品選定の複雑さ

**状況**:
- QNAP製品ラインナップ: **5製品×複数構成** = 数十パターン
- 各製品の違い: CPU性能、メモリ、ネットワーク速度、ベイ数、価格帯
- 顧客要件: 業種、ユーザー数、容量、予算、信頼性要求

**営業担当者の悩み**:
```
「お客様から『30人の会社で10TBのファイルサーバーが欲しい』
 と言われたとき、どの製品を選べばいいのか判断に迷う」

「TS-433eUとTS-h765eUの違いを説明できない」

「RAID 5とRAID 6、どちらを提案すべきか？」
```

**時間のロス**:
- 製品カタログを見比べる: 10分
- 過去の提案資料を探す: 5分
- 上司やベテランに相談: 5分
- **合計: 20分/件**

---

#### 🔴 問題2: 見積もり計算の煩雑さ

**計算が必要な項目**:
1. **NAS本体価格** (製品型番により変動)
2. **HDD/SSD価格** (容量・グレードにより変動)
3. **RAID構成の実効容量計算** (RAID 1は50%、RAID 5は75%など)
4. **保守サービス** (延長保証・センドバック・オンサイト)
5. **導入支援費用** (初期設定・データ移行・教育)
6. **周辺機器** (UPS・ネットワーク機器)
7. **税込み計算**

**実際の作業**:
```
1. Excelシートを開く (または電卓を用意)
2. 製品価格を調べる (エレコムサイトを確認)
3. HDD価格を計算 (容量×枚数×単価)
4. RAID構成を計算 (実効容量の確認)
5. 保守料金を調べる (別資料を参照)
6. 導入支援費を見積もる (工数から計算)
7. 合計を計算 (税抜・税込)
8. 粗利率を確認 (原価を計算)

所要時間: 約30分/件
```

**ミスのリスク**:
- 計算間違い (10件に1件程度)
- 保守料金の見落とし
- 粗利率の計算ミス (赤字見積もりのリスク)

---

#### 🔴 問題3: 提案品質のばらつき

**ベテラン営業の提案**:
- 顧客の業種に応じた最適構成を提案
- ランサムウェア対策やバックアップ戦略も含めた包括提案
- 将来の拡張性も考慮した構成
- **成約率: 40%**

**新人営業の提案**:
- カタログスペックをそのまま提示
- 顧客の課題を十分に理解できていない
- オプション提案が不足
- **成約率: 15%**

**問題の本質**:
```
知識とノウハウが「個人の頭の中」にあり、
組織として共有・標準化されていない
```

---

#### 🔴 問題4: 新人教育のコスト

**現状の教育方法**:
1. 製品カタログの読み込み (1週間)
2. RAID・ネットワークの基礎学習 (2週間)
3. 見積もり作成の練習 (2週間)
4. ベテランの同行営業 (1ヶ月)
5. 独り立ち (3ヶ月目以降)

**教育コスト**:
- 新人の人件費: 3ヶ月間の給与
- ベテランの指導時間: 月20時間 × 3ヶ月
- 機会損失: 新人が独り立ちするまでの期間

**試算**:
```
新人1名の教育コスト = 約150万円
(給与90万円 + ベテラン指導工数60万円)
```

### 2.2 なぜ今、システム化が必要なのか

#### 📈 ビジネス環境の変化

| 要因 | 影響 |
|------|------|
| **DX推進の加速** | 顧客からのNAS導入相談が増加 |
| **ランサムウェア被害の増加** | セキュリティ対策としてのNAS需要増 |
| **リモートワークの定着** | クラウド連携NASへの関心向上 |
| **人手不足** | 営業担当者1人あたりの案件数増加 |

#### ⏰ 緊急性

```
【現状の営業工数】
営業担当者5名 × 月10件 = 月50件の提案
1件あたり110分 = 合計 5,500分/月 (約92時間)

【システム化後の想定】
1件あたり35分 = 合計 1,750分/月 (約29時間)

→ 月63時間の削減 = 営業担当1.5人分の工数
```

**この工数を**:
- 新規顧客開拓に充てる
- 既存顧客のフォローアップに充てる
- より高度な提案活動に充てる

#### 💡 競合他社の動向

- A社: 独自の見積もりシステムを導入済み
- B社: AIを活用した製品推奨システムを開発中

**当社の対応**:
```
今システム化しなければ、
提案スピード・品質で競合に劣後するリスク
```

### 2.3 業務への影響(現状のまま放置した場合)

#### ❌ シナリオ1: 営業機会の損失

```
顧客: 「明日までに見積もりが欲しい」
営業: 「製品選定と見積もり作成に時間がかかり、間に合わない」
結果: 競合他社に案件を取られる

→ 月2件の機会損失 × 平均受注額50万円 = 月100万円の損失
```

#### ❌ シナリオ2: 提案品質の低下

```
新人営業が不適切な構成を提案
→ 顧客が導入後にトラブル発生
→ クレーム対応コストの発生
→ 顧客満足度の低下

→ 年間3件のクレーム × 対応コスト50万円 = 150万円の損失
```

#### ❌ シナリオ3: 営業担当者のモチベーション低下

```
「専門知識の習得に時間がかかり、早く成果を出せない」
「見積もり作成などの事務作業に時間を取られる」
「本来の営業活動に集中できない」

→ 離職率の上昇 → 採用・育成コストの増加
```

### 5.3 製品選定ロジックの説明 (非技術者向け)

#### 選定の基本的な考え方

システムは**3つのステップ**で製品を選定します:

```
【STEP 1】顧客規模の判定
┌────────────────────────────┐
│ ユーザー数と容量から判定    │
│                             │
│ 5人以下 + 5TB以下           │
│ → 小規模 → TS-233          │
│                             │
│ 20人以下 + 20TB以下         │
│ → 中小規模 → TS-433eU      │
│                             │
│ 50人以下 + 50TB以下         │
│ → 中規模 → TS-h765eU       │
│                             │
│ 50人以上 OR 50TB以上        │
│ → 大規模 → TS-H1277AFX     │
└────────────────────────────┘
         ↓
【STEP 2】特殊要件の確認
┌────────────────────────────┐
│ 業種や用途による調整        │
│                             │
│ 医療 OR ランサムウェア対策  │
│ → 高信頼性構成              │
│                             │
│ 動画編集 OR 大容量転送      │
│ → 10GbE対応機種            │
│                             │
│ 監視カメラ                  │
│ → NVR対応機種              │
└────────────────────────────┘
         ↓
【STEP 3】最終調整
┌────────────────────────────┐
│ 予算との照合                │
│                             │
│ 予算内に収まる              │
│ → そのまま推奨              │
│                             │
│ 予算オーバー                │
│ → 容量を下げる              │
│ → または保守を調整          │
└────────────────────────────┘
```

#### 具体例: 製造業30人の場合

```
【入力情報】
業種: 製造業
人数: 30人
容量: 10TB
用途: ファイル共有・CADデータ
予算: 80万円

        ↓

【判定プロセス】

1. 規模判定
   30人 + 10TB → 中規模企業
   → TS-h765eU または TS-433eU が候補

2. 業種判定
   製造業 → CADデータの高速転送が重要
   → 2.5GbE以上推奨
   → TS-h765eU が最適

3. 予算確認
   TS-h765eU (16TB構成) = ¥803,110
   予算80万円 → やや予算オーバー
   → 容量を調整 (12TB構成) → ¥750,000
   → 予算内に収まる

        ↓

【推奨結果】
製品: TS-h765eU
容量: 12TB (4TB×3 + スペア1, RAID 5)
価格: ¥750,000
理由: 中規模製造業に最適、将来の拡張も可能
```

### 5.4 なぜ自動化できるのか?

#### ベテラン営業のノウハウをシステム化

```
【ベテラン営業の頭の中】
┌─────────────────────────────────┐
│ 「製造業で30人規模なら...」       │
│ 「CADデータがあるから高速転送が必要」│
│ 「予算80万円なら TS-h765eU だな」 │
│ 「RAID 5で容量と信頼性のバランスを」│
└─────────────────────────────────┘
            ↓ (システム化)
┌─────────────────────────────────┐
│ IF 製造業 AND 30人 THEN           │
│   製品 = TS-h765eU                │
│                                    │
│ IF CADデータ THEN                 │
│   ネットワーク = 2.5GbE以上       │
│                                    │
│ IF 予算 = 80万円 THEN             │
│   容量調整 = 12TB構成             │
│                                    │
│ IF 容量10TB THEN                  │
│   RAID = RAID 5 推奨              │
└─────────────────────────────────┘
```

**つまり**: ベテランの判断基準を「IF-THEN」のルールに落とし込むことで、誰でも同じレベルの提案が可能になります。

### 8.1 WEB2とは何か?

#### 基本コンセプト

```

┌────────────────────────────┐
│  ✓ 基本的な提案支援機能     │
│  ✓ 個人での利用             │

            ↓ 拡張
【WEB2】動的管理システム
┌────────────────────────────┐
│  ✓ 高度な管理機能           │
│  ✓ チーム全体で利用         │
│  ✓ データベースで一元管理   │
│  ✓ 製品情報を動的更新       │
│  ✓ 案件管理・レポート機能   │
└────────────────────────────┘
```

#### WEB1とWEB2の違い

| 項目 | WEB1 (現在) | WEB2 (将来) |
|------|------------|------------|
| **データ保存** | ブラウザ内 | データベース |
| **製品マスタ** | 手動編集 | 管理画面で編集 |
| **見積もり履歴** | なし | 検索・再利用可能 |
| **チーム共有** | 不可 | 可能 |
| **案件管理** | なし | 進捗管理機能 |
| **レポート** | なし | 月次・年次レポート |
| **ユーザー管理** | なし | ログイン・権限管理 |
| **価格更新** | 手動 | 自動取得 (将来) |

### 8.2 なぜWEB2が必要なのか?

#### 現状の課題

```
【課題1】データの散逸
┌────────────────────────────┐

【課題2】製品情報の更新が手間
毎回HTMLファイルを編集・アップロード
→ 更新ミスのリスク

【課題3】案件の進捗が見えない
どの営業が、どの案件を、

「今月何件提案したか?」
```

```
管理者が管理画面から直接編集
→ 即座に全営業担当に反映

【実現3】案件管理
各案件の進捗をリアルタイムで共有
→ チーム全体で状況を把握

【実現4】レポート・分析

→ 経営判断に活用
```

#### Q18: システムが使えなくなったらどうしますか?

**A**: 複数の対策を用意しています。

```
【バックアップ体制】

   └ WEB1との並行運用期間中はWEB1を使用
   └ WEB2単独運用後は緊急連絡フロー確立

   └ 毎日自動バックアップ
   └ 異なる場所に保管 (冗長化)

   └ 最大4時間以内に復旧
   └ 復旧できない場合はオフライン対応

従来の方法 (Excel等) で業務継続可能
→ 業務が止まることはありません
```

### 用語集

| 用語 | 説明 |
|------|------|
| **RAID** | 複数のHDDを組み合わせてデータを保護する技術。RAID 1はミラーリング、RAID 5/6はパリティによる冗長化。 |
| **NAS** | Network Attached Storageの略。ネットワークに接続して複数ユーザーで共有できるストレージ。 |
| **ベイ** | HDDを搭載するスロット。4ベイなら4台のHDDを搭載可能。 |
| **2.5GbE** | 2.5ギガビットイーサネット。1GbEの2.5倍の通信速度。 |
| **JSON** | データを保存する形式の一つ。テキスト形式で人間にも読みやすい。 |
| **MVP** | Minimum Viable Productの略。最小限の機能で価値を提供する製品。 |
| **ROI** | Return On Investmentの略。投資対効果を示す指標。 |
| **API** | Application Programming Interfaceの略。システム間でデータをやり取りする仕組み。 |
